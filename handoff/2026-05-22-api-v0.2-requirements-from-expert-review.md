# API v0.2 Requirements From Expert Review

## Purpose

This is the required confirmation checklist for moving the pre-sync API v0.2
draft into a confirmed / frozen API v0.2 after expert review and the Thursday
sync. Use it after the Thursday sync to update the API document with confirmed
慧誠 field names.

Current freeze judgment: v0.2 can move forward as the working engineering
contract, but it is not frozen until the owner/date/fallback items below are
closed.

Post-sync update on `2026-05-21`: the June workflow default is now
`post_measurement_only`. Endpoint 1 and Endpoint 3 should be merged for the
first June integration pass. The separate vitals-ready endpoint remains an
optimized future workflow, not the current freeze blocker.

Current June contract summary:

```text
POST /api/triage-demo/sessions
  Request: measured or synthetic iMVS vital payload
  Response: session_key + first typed question

POST /api/triage-demo/sessions/{session_key}/answers
  Request: session_key + answer payload
  Response: next typed question or staff_review_summary
```

Freeze blockers:

- iMVS field names, units, required/optional status, and missing/failure
  representation.
- Post-measurement UI insertion point and report/result display behavior.
- Generic iMVS question-template support: `single_choice`, `multi_choice`,
  numeric / scale, variable option count, maximum option count, label length,
  and no-scroll behavior.
- First case-lane selection: respiratory synthetic, tachycardia live-performance,
  or healthy/unhealthy contrast.
- Staff-summary wording approval.
- Demo endpoint/security/log/screenshot/no-PHI boundary.
- Remote REST API Mode plus clearly labeled Local Scripted Demo Mode rehearsal
  date.

## Scope Decision

API v0.2 must continue to support only:

```text
synthetic iMVS vital payload
-> structured questions
-> answer loop
-> staff_review_summary
```

After the imedtac engineering sync, v0.2 should use this June flow:

```text
iMVS completes measurement
-> measured vital payload starts NYCU session
-> structured answer loop
-> staff_review_summary
```

The two-phase flow remains an optimized future path after the post-measurement
integration loop is stable.

It must not introduce:

- diagnosis;
- final triage / acuity level;
- treatment advice;
- emergency order;
- production HIS / EMR / FHIR writeback;
- real patient data;
- raw ASR audio.

## Required Additions

### Question Template Rendering

Post-Duobao internal sync requirement: v0.2 must make the UI contract explicit
enough that imedtac can render a question from metadata rather than hand-coding
each screen.

Add or confirm:

- `question.type`
- `question.ui_template`
- `question.option_count`
- `question.options[].id`
- `question.options[].label`
- `question.none_option_id`
- `question.scale.min`
- `question.scale.max`
- `question.scale.min_label`
- `question.scale.max_label`
- `question.rendering_constraints.requires_no_scroll`
- `question.rendering_constraints.max_visible_options_without_scroll`
- `capabilities.max_options_per_question`
- `capabilities.max_option_label_length`
- `capabilities.variable_option_count`

Freeze rule:

```text
If iMVS cannot render generic typed question templates, freeze API v0.2 as
backend/sample-JSON only and use a local scripted UI for the June demo.
```

### Session

Add:

- `session_expires_at`
- `session_state`
- `last_question_id`

Allowed `session_state` values:

- `active`
- `summary_ready`
- `expired`
- `abandoned`
- `error`

### Retry And Idempotency

Add:

- `request_id`
- `idempotency_key`
- response echo of `request_id`
- `response_id`

For demo, these can be stable UUID-like strings. They prevent duplicate answer
submission from advancing the flow twice.

Required idempotency behavior:

```text
Same endpoint + same session_key + same idempotency_key + same request body
must return the same result and must not advance the question flow twice.
```

If the same idempotency key arrives with a materially different body, return
`error.code=idempotency_conflict` and do not advance the flow.

### Versioned Clinical Content

Add:

- `case_version`
- `fixture_version`
- `question_set_version`
- `wording_version`

Reason: `case_id` identifies the synthetic case family, but case content,
question wording/order, and staff-summary wording can change clinical meaning.
`data/version_manifest.json` is the current control point for these values.

### Vital Payload Quality

Add per session:

- `measurement_timestamp`
- `device_id`

Add per vital field when possible:

- `measurement_status`: `measured`, `missing`, `failed`, `manual_entry`, or
  `not_available`
- `quality_flag`: `ok`, `needs_review`, `device_error`, `out_of_range_demo`, or
  `unknown`
- `missing_reason`

Current v0.2 examples use per-vital quality objects. If 慧誠 cannot provide
per-vital quality by June, the document must explicitly mark session-level
quality as provisional and move per-vital quality to v0.3.

### June Workflow Mode

Post-sync status: `post_measurement_only` is the June default and the current
freeze target.

Required June values:

- `workflow_mode`: `post_measurement_only`
- `measurement_state`: `complete`
- `vitals_ready`: `true`
- `question_phase`: `post_measurement_intake` for question responses, or
  `summary` for summary responses

Required June endpoint shape:

```text
POST /api/triage-demo/sessions
POST /api/triage-demo/sessions/{session_key}/answers
```

`POST /api/triage-demo/sessions` must include the measured or synthetic vital
payload in the first request. It returns `session_key` and the first typed
question. `POST /api/triage-demo/sessions/{session_key}/answers` returns either
the next typed question or `staff_review_summary`.

### Future Two-Phase Workflow

Post-sync status: optimized future path, not June default.

Keep these fields when the API needs to represent both current and future
workflow modes:

- `workflow_mode`: `parallel_measurement_intake` or `post_measurement_only`
- `measurement_state`: `not_started`, `in_progress`, `complete`, or `failed`
- `vitals_ready`: boolean
- `safe_to_ask_phase1_question`: boolean when `measurement_state=in_progress`
- `question_phase`: `post_measurement_intake` for the June default,
  `pre_vital_intake` / `post_vital_followup` for future two-phase mode, or
  `summary`
- `phase_reason`: short explanation for why the current question belongs in
  the current phase

Recommended additional endpoint for optimized flow:

```http
POST /api/triage-demo/sessions/{session_key}/vitals
```

Use it only when Phase 1 starts before vital measurement is complete. It is not
a June freeze blocker unless a later stakeholder decision explicitly reopens the
two-phase path.

### Output Control

Add:

- `summary_visibility: "staff_only"`
- `handoff_required`
- `handoff_reason_codes`
- `staff_handoff_note`

Replace:

- `plan_support` -> `review_action`
- `assessment_support` -> `review_basis`

Reason: `plan_support` may be read as a SOAP Plan / medical action. For June
demo, the output should only tell staff or clinicians to review the measured
vitals and reported symptoms. `assessment_support` can also read too close to a
SOAP Assessment field; use `review_basis` unless 多寶 explicitly asks for a
different approved label.

### Error

Every error must include:

- `status: "error"`
- stable `error.code`
- human-readable `error.message`
- `retry_allowed`
- no fake summary

Recommended error codes:

- `missing_required_field`
- `unsupported_question_type`
- `invalid_session`
- `session_expired`
- `api_timeout`
- `measurement_quality_unavailable`
- `idempotency_conflict`

Every error must include:

- `http_status`
- `fallback_to_standard_staff_workflow: true`
- no `staff_review_summary`
- no fabricated clinical output

Minimum demo examples now required:

- invalid session
- missing required field
- unsupported question type
- measurement quality unavailable
- API timeout

### Question Traceability

Every runtime API question ID must map to registry question IDs before v0.2 can
freeze. Current mapping lives in `../data/api_question_mapping.csv`.

Example shape:

```json
{
  "question_id": "breathing-duration",
  "registry_refs": ["RESP-002"],
  "source_refs": ["CDC-FLU-WARN", "LOCAL-PROTOCOL-TBD"],
  "evidence_status": "source-family",
  "review_owner": "clinical_reviewer_tbd"
}
```

The first respiratory flow is now registered as:

```text
FLOW-RESPIRATORY-EARLY-HANDOFF v0.2
```

## First Respiratory Flow

The respiratory flow should be early-handoff, not full eight-question intake:

1. Start session after measured vitals are available.
2. Question 1: chief complaint.
3. Question 2: dyspnea duration / patient-reported severity.
4. Question 3: chest pain / pressure.
5. Question 4: chronic lung disease / baseline oxygen / medication
   context.
6. Staff-review summary.

## Runtime Enforcement Checklist

Before sharing a demo after expert review, check:

- [ ] No `diagnosis` output field.
- [ ] No `plan_support` field.
- [ ] No `assessment_support` field unless a named clinical owner explicitly
      approves that exact API label.
- [ ] No final acuity / ESI output.
- [ ] No treatment advice.
- [ ] No safe-to-go-home wording.
- [ ] No FDA-cleared / 510(k)-ready claim.
- [ ] No real name, MRN, national ID, phone, address, raw audio, or real chart
      content in payloads or logs.
- [ ] API errors do not generate summary text.
- [ ] `summary_visibility` is `staff_only`.
- [ ] `handoff_required` is explicit for respiratory case.
- [ ] June requests use `workflow_mode=post_measurement_only`,
      `measurement_state=complete`, and `vitals_ready=true`.
- [ ] June start-session requests include measured or synthetic vital payloads.
- [ ] June answer responses return either the next typed question or
      `staff_review_summary`.
- [ ] Future two-phase examples are clearly labeled as future mode and not as
      required June integration.
- [ ] Runtime question IDs are mapped in `data/api_question_mapping.csv`.
- [ ] `case_version`, `fixture_version`, `question_set_version`, and
      `wording_version` match `data/version_manifest.json`.
